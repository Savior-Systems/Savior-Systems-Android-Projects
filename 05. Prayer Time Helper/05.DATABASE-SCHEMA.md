# 05. Database Schema & DataStore — Prayer Time Helper

While the `Adhan` library calculates times on the fly quickly, caching a week's worth of timings in Room is highly recommended. This ensures immediate UI rendering and allows the `AlarmManager` to easily query upcoming times without spinning up the calculation engine repeatedly in the background.

---

## 1. Room Database Schema

### Table: `prayer_cache`
Stores calculated prayer timings. We cache the current day + 7 days into the future.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `date_id` | String | **PK**, Not Null | Format: `YYYY-MM-DD`. Used for fast lookups. |
| `latitude` | Double | Not Null | Location used for this calculation. |
| `longitude`| Double | Not Null | Location used for this calculation. |
| `fajr` | Long | Not Null | Unix timestamp (milliseconds) |
| `sunrise` | Long | Not Null | Unix timestamp |
| `dhuhr` | Long | Not Null | Unix timestamp |
| `asr` | Long | Not Null | Unix timestamp |
| `maghrib` | Long | Not Null | Unix timestamp |
| `isha` | Long | Not Null | Unix timestamp |
| `hijri_date`| String | Not Null | Pre-calculated Hijri date string (e.g., "14 Ramadan 1445") |

---

## 2. Room DAO Queries

```kotlin
@Dao
interface PrayerCacheDao {

    // Fetch timings for a specific day
    @Query("SELECT * FROM prayer_cache WHERE date_id = :dateId")
    suspend fun getTimingsForDate(dateId: String): DailyPrayerEntity?

    // Fetch the next 7 days (used by WorkManager to set alarms)
    @Query("SELECT * FROM prayer_cache WHERE date_id >= :todayId ORDER BY date_id ASC LIMIT 7")
    suspend fun getUpcomingWeek(todayId: String): List<DailyPrayerEntity>

    // Insert or replace calculated timings
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertTimings(timings: List<DailyPrayerEntity>)

    // Clear cache (called when location or calculation method changes)
    @Query("DELETE FROM prayer_cache")
    suspend fun clearCache()
    
    // Delete past dates to prevent database bloat
    @Query("DELETE FROM prayer_cache WHERE date_id < :todayId")
    suspend fun deletePastRecords(todayId: String)
}
```

## 3. DataStore Preferences (Settings)

All user configurations are stored in `DataStore<Preferences>`.

| Preference Key | Type | Default Value | Description |
| :--- | :--- | :--- | :--- |
| `pref_lat` | Double | `0.0` | Saved latitude. |
| `pref_lng` | Double | `0.0` | Saved longitude. |
| `pref_city_name` | String | `""` | User-friendly city name (e.g., "Dhaka, BD") |
| `pref_calc_method` | String | `"KARACHI"` | Adhan `CalculationMethod` enum string. |
| `pref_madhab_asr` | String | `"HANAFI"` | Either "HANAFI" or "SHAFI". |
| `pref_high_lat` | String | `"NONE"` | Rule for extreme latitudes (None, NightMiddle, AngleBased). |
| `pref_hijri_adj` | Int | `0` | Manual adjustment for Hijri calendar (-2 to +2 days). |

### Alarm Configurations (Stored as JSON string or individual boolean keys)

## 4. Cache Invalidation Logic
The Room database `prayer_cache` must be wiped and recalculated if any of the following DataStore preferences change:
1.  `pref_lat` / `pref_lng`
2.  `pref_calc_method`
3.  `pref_madhab_asr`
4.  `pref_high_lat`
