# 08. Play Console Policy & Safety — FocusPulse Timer
### Version 2.0 · Growth Edition

---

## 1. Permissions Specification

FocusPulse Timer follows a **Zero-Permission-by-Default** architecture. The table below documents every permission and its justification.

| Permission | API Level | Required? | Justification | Declared in Manifest? |
|:-----------|:---------:|:---------:|:--------------|:---------------------:|
| `POST_NOTIFICATIONS` | API 33+ (Android 13+) | Yes | Required to show timer completion alerts when app is backgrounded. Requested at runtime on first session start. | ✅ |
| `FOREGROUND_SERVICE` | All | Yes | Timer runs as a foreground service with a persistent notification so it survives system reclaim. | ✅ |
| `FOREGROUND_SERVICE_MEDIA_PLAYBACK` | API 34+ | Conditional | Required if soundscape playback is attached to foreground service on Android 14+. | ✅ |
| `RECEIVE_BOOT_COMPLETED` | All | Yes | Required to reschedule streak-check WorkManager task after device reboot. | ✅ |
| No other permissions | — | — | No location, camera, microphone, contacts, file system, or network (beyond AdMob/Firebase SDK) | — |

### Permission Request Flow
```
[First session start]
      │
      ▼
Check Build.VERSION.SDK_INT >= 33?
      │
      ├── Yes → ActivityResultContracts.RequestPermission(POST_NOTIFICATIONS)
      │               │
      │               ├── Granted → start foreground service normally
      │               └── Denied → timer still works, notification won't appear;
      │                             show a one-time in-app tip "Enable notifications
      │                             to get timer alerts"
      │
      └── No (API < 33) → POST_NOTIFICATIONS not required; proceed normally
```

## 2. FileProvider Configuration (Focus Card Sharing)

The Focus Card share feature uses `FileProvider` to share bitmap images WITHOUT requiring `WRITE_EXTERNAL_STORAGE` permission (which is restricted on API 29+).

**AndroidManifest.xml entry**:
```xml
<provider
    android:name="androidx.core.content.FileProvider"
    android:authorities="${applicationId}.provider"
    android:exported="false"
    android:grantUriPermissions="true">
    <meta-data
        android:name="android.support.FILE_PROVIDER_PATHS"
        android:resource="@xml/file_paths" />
</provider>
```

**`res/xml/file_paths.xml`**:
```xml
<?xml version="1.0" encoding="utf-8"?>
<paths>
    <cache-path name="focus_cards" path="." />
</paths>
```

*Focus cards are written to `cacheDir` (not external storage) and shared via `FLAG_GRANT_READ_URI_PERMISSION`. The file is overwritten on each new card. No storage permission is needed.*

## 3. Glance Widget — No New Permissions Required

The Jetpack Glance home screen widget uses the standard `AppWidget` framework:
*   `android.appwidget.action.APPWIDGET_UPDATE` handled by `FocusWidgetReceiver` — no special permission.
*   Widget reads timer state from in-memory state shared via `GlanceStateDefinition` — no IPC permission.
*   Widget actions use `actionRunCallback<ToggleTimerAction>()` which routes to a Hilt-injected use case — no `BROADCAST_STICKY` or other permissions.

**`AndroidManifest.xml` widget declaration**:
```xml
<receiver
    android:name=".ui.widget.FocusWidgetReceiver"
    android:exported="true">
    <intent-filter>
        <action android:name="android.appwidget.action.APPWIDGET_UPDATE" />
    </intent-filter>
    <meta-data
        android:name="android.appwidget.provider"
        android:resource="@xml/focus_widget_info" />
</receiver>
```

## 4. Data Safety Form — Play Console Declarations

### Data Collected and Shared

| Data Type | Collected? | Purpose | Shared with 3rd Party? | User Can Delete? |
|:----------|:----------:|:--------|:----------------------:|:----------------:|
| Device ID (AdMob) | Yes (automatic) | Advertising / Analytics (AdMob SDK) | Yes — Google AdMob | Via AdMob Privacy Console |
| Firebase Crash Logs | Yes (anonymous) | App stability diagnostics | Yes — Google Firebase | N/A (anonymous) |
| Firebase Analytics Events | Yes (anonymous) | Understanding app usage patterns | Yes — Google Firebase | N/A (anonymous) |
| Study session data | **No** | Stored locally only on device (Room DB) | **No** | Yes — "Clear History" in Settings |
| Personal information | **No** | Not collected at any point | **No** | — |
| Location | **No** | Not required | **No** | — |

### Encryption
*   Data transmitted to AdMob and Firebase is **encrypted in transit** (HTTPS/TLS).
*   Local Room Database data is NOT end-to-end encrypted (no sensitive personal data is stored).

### Privacy Policy
*   Must host a unique privacy policy at: `https://saviorsystems.github.io/privacy/focuspulse`
*   Policy must detail: AdMob data collection, Firebase event logging, no personal data collection, data deletion instructions.
*   Link must be entered in Play Console → App Content → Privacy Policy.

## 5. AdMob & UMP GDPR Consent

UMP (User Messaging Platform) consent must be shown on first launch for EEA/UK users:

```kotlin
// In MainActivity.onCreate()
val params = ConsentRequestParameters.Builder()
    .setTagForUnderAgeOfConsent(false)
    .build()

ConsentInformation.getInstance(this).requestConsentInfoUpdate(
    this, params,
    {
        if (ConsentInformation.getInstance(this).isConsentFormAvailable) {
            loadAndShowForm(this)
        } else {
            // Consent not required (outside EEA) — proceed to load ads
            initializeAdMob()
        }
    },
    { _ -> initializeAdMob() } // On error — proceed to load ads
)
```

*All ad loading must be gated behind UMP consent resolution. Never load ads before consent check completes.*
