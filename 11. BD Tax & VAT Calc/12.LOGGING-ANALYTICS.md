# 12. Logging & Analytics Schema — BD Tax & VAT Calc

Telemetry tracking definitions monitoring calculations performance while protecting user data privacy.

---

## 1. Firebase Custom Events Schema

We track 4 custom analytics events to analyze user workflows:

### Event A: `tax_calculation_run`
*   **Trigger**: User clicks "Calculate" on the tax tab.
*   **Parameters**:
    *   `tax_year` (string: "2025-2026", "2026-2027")
    *   `exemption_tier` (string: "general", "women", "disabled", "freedom_fighters")
    *   `minimum_tax_applied` (integer: 1 or 0)

### Event B: `vat_calculation_run`
*   **Trigger**: User calculates VAT amounts.
*   **Parameters**:
    *   `vat_mode` (string: "inclusive", "exclusive")
    *   `vat_rate_preset` (double: e.g. 15.0, 7.5, 5.0)

### Event C: `calculation_saved`
*   **Trigger**: User logs a calculation record to local Room database.
*   **Parameters**:
    *   `entry_type` (string: "TAX", "VAT")

### Event D: `cooldown_ad_suppressed`
*   **Trigger**: Interstitial ad request is suppressed because of the 180s cooldown rule.
*   **Parameters**:
    *   `elapsed_time_sec` (integer)

## 2. Privacy Guardrails ("NEVER Log" Rules)

Strict guidelines to protect client identity:
*   🚫 **NEVER log Raw Salary Amounts**: Salary inputs (gross salary, house rent, basic salary) must never be logged. Only aggregate results buckets are permitted if needed.
*   🚫 **NEVER log Calculation Titles**: Custom titles saved by the user (e.g. "Sajjad Invoice 104") are strictly forbidden in parameters.
*   🚫 **NEVER log Document Details**: The values of calculated fields must stay inside the local sandboxed database and never cross the device boundary.
*   🚫 **NEVER log Location/IP**: Network tracing coordinates must be stripped.
